# 楽天ランキングAPI × GAS 連携システム

楽天市場の商品ランキングAPIを使用して、指定された複数カテゴリ（ジャンル）のランキング情報を定期的に取得し、Google スプレッドシートへ自動保存するGoogle Apps Script (GAS) システムです。

---

## 1. システム概要・ファイル構成

本プロジェクトは、機能フェーズごとにファイルを分割し、保守性と独立性を担保した設計を行っています。

```
ItemRankingAPI/
├── Phase 1_基盤構築スクリプト.gs       # 認証プロパティ取得、設定シート初期化、API接続の低レベル関数
├── Phase_2_データ取得・書き込みスクリプト.gs # 設定読み込み、データ整形（ソート）、スプレッドシート書き込みメインバッチ
└── Phase_3_運用・安定化スクリプト.gs       # 自動リトライ、実行ログの記録、時間主導型トリガー自動設定
```

---

## 2. 初期設定手順

本システムを実行する前に、以下の準備が必要です。

### 2.1. スクリプトプロパティの登録
GASエディタの「プロジェクトの設定」（歯車アイコン ⚙️）から、以下のスクリプトプロパティを登録してください。

| プロパティ名（キー） | 設定する値 |
| :--- | :--- |
| `RAKUTEN_APPLICATION_ID` | 楽天APIのアプリケーションID |
| `RAKUTEN_ACCESS_KEY` | 楽天APIのアクセスキー |
| `TARGET_SPREADSHEET_ID` | 書き込み対象となるスプレッドシートのID（URLの `/d/【ここ】/edit` の部分） |
| `RAKUTEN_APP_URL` | 楽天デベロッパーのアプリ管理で登録した「アプリURL」（例: `https://script.google.com/`） |

### 2.2. 「設定」シートの作成
スプレッドシート側で手動作成するか、GASエディタで `initializeSettingSheet` 関数（またはテスト用の `testInitializeSettingSheet`）を実行して、設定用のシートを自動生成してください。

**「設定」シートのレイアウト:**
* **A列（ジャンル名）**: スイーツ などの管理名
* **B列（ジャンルID）**: 楽天のジャンルID（例: スイーツは `100283`）
* **C列（取得件数）**: 取得したい件数（例: `30`）
* **D列（有効フラグ）**: `TRUE`（有効） または `FALSE`（無効）

## 3. 【重要】API疎通テストでの課題と解決への試行錯誤

本システムの構築にあたり、**ステップ 1-3（API疎通テスト）における403エラーの回避** と、**楽天APIの応答データ構造のブレへの対処** で大きな試行錯誤がありました。以下にその経緯と解決策を詳しく記載します。

### 3.1. 403エラー（リファラ制限）の解消プロセス
楽天デベロッパーAPIにはセキュリティ向上のため、リクエスト送信元のドメイン（リファラ）を厳格にチェックする仕様があります。

#### 【第1段階】リファラの欠落（403: `REQUEST_CONTEXT_BODY_HTTP_REFERRER_MISSING`）
* **発生事象**: `UrlFetchApp.fetch()` を使って普通にリクエストを送ると、楽天APIから「リファラが存在しない」というエラーが返却されました。
* **原因**: GASのサーバーサイド通信である `UrlFetchApp` は、デフォルトではHTTPヘッダーに `Referer` を付与して送信しないためです。

#### 【第2段階】ダミーリファラでの試行と拒否（403: `HTTP_REFERRER_NOT_ALLOWED`）
* **発生事象**: リクエストヘッダーに一般的な `Referer: https://www.rakuten.co.jp/` やダミーのURLを指定してリクエストを投げたところ、エラーが「許可されていないリファラである」に変化して拒否されました。
* **原因**: 楽天デベロッパーのアプリ登録時に設定した「アプリケーションURL」（例: `https://script.google.com/`）以外のドメインからのアクセスを、API側が完全にブロックしているためでした。

#### 【第3段階】解決：`Referer` と `Origin` の同時注入
* **対応策**: 楽天側に登録した正規のアプリケーションURLである `https://script.google.com/` をスクリプトプロパティ `RAKUTEN_APP_URL` から動的に読み込み、リクエストの **`Referer` ヘッダー** と **`Origin` ヘッダー** の双方に明示的に設定して送信しました。
* **結果**: APIの認証システムが「登録済みの正規送信元からのリクエスト」と判定し、403エラーをクリアすることに成功しました。

### 3.2. レスポンスJSON形式の非一貫性への対処
APIテストフォームから取得されたデータ構造を分析した結果、楽天APIは呼び出し元の設定やパラメータによって、取得できる商品オブジェクトの階層構造が変化する性質があることが判明しました。

* **階層A (ネスト構造)**: `response.Items[i].Item.itemName`
* **階層B (フラット構造)**: `response.Items[i].itemName`

#### 対策：自動正規化マッパーの実装
この構造のブレに対応するため、取得処理を行う内部関数 `fetchRakutenRanking_` の中で、各商品データに対し `Item` というキー（ネスト）が存在するかどうかを自動判定するマッパーを記述しました。これにより、どちらの構造であってもプログラムがクラッシュすることなく、常に同じフラットなオブジェクト配列 `[{ rank, itemName, itemPrice, ... }]` として次の処理（Phase 2）へ引き渡すよう堅牢化しました。

## 4. テスト一覧と実行方法

各機能は、GASエディタ上のプルダウンメニューから以下のテスト関数を選択し、「実行」をクリックすることで検証できます。

| 検証フェーズ | テスト関数名 | テスト内容 | 安全性への配慮 |
| :--- | :--- | :--- | :--- |
| **Phase 1** | `testInitializeSettingSheet` | 設定シート自動作成の検証 | `'設定_TEST'` という一時シートを使い、検証後に自動削除します。 |
| | `testRakutenApiConnection` | 楽天APIとの直接の通信疎通テスト | 1件のみ試しに取得し、ログに出力します。 |
| **Phase 2** | `testReadSettings` | 設定シートからの有効設定読み込み検証 | 有効フラグが `TRUE` の行のみ正常に読み込めるか検証します。 |
| | `testFetchAndTransform` | API連携とソート、スライス検証 | API取得データを順位昇順（1位〜）にソート・スライスできているか検証します。 |
| | `testExportRankings` | **結合テスト（全機能実行）** | 実際に設定を読み込み、対象シート（例: 「スイーツ」）へ昇順で書き込みます。 |
| **Phase 3** | `testApiRetryBehavior` | 自動リトライ動作検証 | 503エラーや通信例外時に3回リトライが正常に走るかをログで追跡します。 |
| | `testWriteLog` | 実行ログ記録機能の検証 | 「実行ログ」シートに成否ログが2行（SUCCESS/ERROR）追記されるか検証します。 |
| | `testTriggerSetup` | 定期トリガー設定・削除検証 | トリガー設定後、正常に削除されクリーンアップされるかを検証します。 |

---

## 5. 運用方法（定期起動の設定）

テストがすべて完了した後は、以下の手順で時間指定トリガーを設定することで完全自動運用が開始されます。

1. GASエディタの上部実行関数から **`setDailyTrigger`** を選択し、実行します。
2. これにより、毎日午前9時〜10時の時間帯にバッチ（`exportRakutenRankings`）が自動で起動するよう設定されます。
3. 実行時間は、`setDailyTrigger(12)` のように引数に時間を渡して実行することでいつでも変更可能です。
4. トリガーを止めたい場合は、**`deleteDailyTrigger`** を実行するか、GASエディタ左メニューの「トリガー（⏰）」から手動で削除してください。
